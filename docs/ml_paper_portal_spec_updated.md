# 機械学習論文 日本語入口サイト 仕様メモ

## 1. 企画の核
このサイトは、**機械学習系の無料で読める論文に対して、日本語の入口を作るサイト**。

目的は「論文の全文を日本語化すること」ではなく、  
**ユーザーが“読む価値があるか・興味があるか”を短時間で判断できるようにすること**。

イメージとしては、

- 小説の帯
- 辞書をぱらぱら眺める感覚
- wiki的に関連項目を辿れる入口

に近い。

---

## 2. サイトの役割
役割はあくまで**入口**であり、全文解説や翻訳代替ではない。

役割分担は以下。

- サイト側: 日本語の短い導入、タグ整理、関連導線
- ユーザー側: 気になった論文を開く
- 深読み: ユーザーがPDFをLLM等に入れて読む

そのため、各ページは長文ではなく、**短く・判断しやすく・回遊しやすく**作る。

---

## 3. 想定分野
対象はまず **機械学習系** に絞る。

理由:
- 更新が速く、新着需要がある
- 英語論文が多く、日本語入口の価値が高い
- パラパラ眺める需要がある
- タグ設計と相性が良い

初期対象候補:
- LLM
- Vision
- Multimodal
- Diffusion
- Agent
- 超解像
- 画像分類
- 異常検知

---

## 4. 価値の出し方
このサイトは「有用論文だけを厳選する場」ではなく、  
**“興味を引く場”** を目指す。

重要なのは、

- 有用かどうか
- 正しいかどうか

だけではなく、

- 面白そうか
- 自分の関心に近いか
- なんとなく開いてみたいか

を生むこと。

つまり、
**評価の代行**ではなく、**興味の着火**が主目的。

---

## 5. データソース
主軸は **OpenAlex**。

必要に応じて arXiv を補助利用する可能性あり。

想定する取得情報:
- 論文タイトル
- 著者
- 公開日
- 被引用数
- Open Access 情報
- 論文URL / PDF URL
- トピック / 分野情報
- 抄録素材

方針:
- 無料で読める論文中心
- Open Access 優先
- 取得はAPI経由
- 日次または数回/日で巡回

---

## 6. ページ構成方針

### 6-1. 論文ページ
論文ページの基本構成は以下。

1. タイトル
2. 超ざっくりまとめ
3. キーワード
4. 少し長めの要約
5. 元論文情報
6. 関連タグ / 関連論文

### 6-2. 冒頭の超ざっくりまとめ
最重要。  
ここは「帯」や「あらすじ」に相当する。

入れる内容:
- 何の論文か
- 何が新しいか
- どんな人に刺さるか

### 6-3. キーワード
冒頭直下に配置。  
ぱっと見で分野がわかるようにする。

例:
- LLM
- Vision
- Diffusion
- ViT
- 軽量化
- 超解像

### 6-4. 少し長めの要約
本文は長すぎず、入口として十分な長さに抑える。

内容候補:
- 背景
- 提案
- 面白い点
- 向いている読者

### 6-5. 元論文情報
信頼性の核。

表示候補:
- 原題
- 著者
- 公開日
- arXiv / DOI / OpenAlexリンク
- PDFリンク

---

## 7. タイトル / 文面の方向性
タイトルは検索結果で押したくなる形を重視する。

候補方向:
- 〇〇を改善する機械学習論文を日本語でざっくり整理
- 【機械学習論文】〇〇とは？ 何が新しいのかを短く把握
- 〇〇の新手法 - 元論文リンク付きで日本語要点整理

狙い:
- わかりやすそう
- 飛ばし記事っぽくない
- 元論文ベースで信頼できそう
- 軽く読めそう

---

## 8. 内部リンク方針
### 8-1. 元祖論文自動推定はやらない
完全自動で「元祖論文」を決めるのは危険なので採用しない。

理由:
- 元祖の定義が難しい
- 間違えると信頼を落とす
- 表記ゆれや先行研究の問題がある

### 8-2. 本文中のモデル名・手法名はタグページへ
本文中の語句リンクは、**元祖論文ではなくタグページへ飛ばす**。

例:
- ViT → ViTタグページ
- Diffusion → Diffusionタグページ
- 画像分類 → 画像分類タグページ

役割:
- 概念のハブにする
- 安全に回遊導線を作る
- 関連論文を一覧で見せる

---

## 9. タグ設計
タグは階層っぽく見せるが、内部的には柔軟な多対多を前提にする。

### 9-1. 階層案
#### 大タグ
分野・問題設定・用途

例:
- 画像認識
- LLM
- 超解像
- 異常検知
- 強化学習

#### 中タグ
モデル系列・手法群

例:
- ViT
- CNN
- Diffusion
- Transformer
- GNN
- Agent

#### 小タグ
具体的な技術要素

例:
- 軽量化
- 蒸留
- 量子化
- few-shot
- 推論高速化
- マルチモーダル

### 9-2. 組み合わせ検索
タグは将来的に組み合わせ検索できるようにする。

例:
- 画像認識 × ViT
- LLM × Agent
- 超解像 × Diffusion
- 画像認識 × ViT × 軽量化

---

## 10. タグページ方針
タグページは一覧ページであると同時に、軽い入口解説も持つ。

### 10-1. 構成
1. タグ名
2. 件数や関連タグなどの補助情報
3. 「このタグとは？」トグル
4. 論文一覧
5. 関連タグ

### 10-2. タグ解説
タグページに「〇〇とは？」の超簡単解説を置く。

例:
- 画像分類とは？
- ViTとは？
- 超解像とは？

ただし、**デフォルトは閉じる**。

理由:
- 意味を知りたい人には役立つ
- 既に知っている人には邪魔になる
- 一覧をすぐ見たい人の導線を邪魔しない

### 10-3. タグページに置きたいもの
- このタグとは？（トグル）
- よく関連するタグ
- 代表的な論文
- 最近の論文
- 並び替え（最新順 / 被引用数順 など）

※ 「元祖論文」ではなく「代表的な論文」「まず見るなら」を使う。

---

## 11. 回遊設計
このサイトは単発閲覧ではなく、**辿っていける構造**が大事。

理想の流れ:
1. 検索で個別論文ページに来る
2. タグが気になる
3. タグページへ飛ぶ
4. 関連論文を見る
5. 別の論文へ移る

wiki感に近い回遊を狙う。

---

## 12. トップページ方針
高度なおすすめは最初から入れない。

代わりに、見やすい導線として以下を置く。

- 今日のおすすめ
- 最新記事
- 人気タグ
- タグから探す

### 12-1. 今日のおすすめ
「おすすめ」は高度レコメンドではなく、**軽いランダムピック**でよい。

例:
- 直近30日以内からランダム
- タグが偏りすぎないようにする
- 古すぎるものは除外

狙い:
- 毎回少し違うものが見える
- ぱらぱら眺めやすい
- サイトが生きている感じが出る

## 13. 運用方針
### 13-1. 初期
- 新着取得
- 要約生成
- 管理画面で承認
- 公開

この段階では半自動。

### 13-2. 安定後
- 取得自動
- 要約自動
- 公開自動
- 失敗時だけ通知
- API仕様変更や品質劣化時のみ介入

理想は
**普段は自走、変化点だけ介入**。

## 16. 取得論文の配分ルール
拾ってくる論文は、**被引用数が多いもの** と **それ以外の新しめ・埋もれ気味のもの** を **半々程度** で混ぜる方針にする。

### 16-1. 目的
- 定番・代表的な論文ばかりに偏らせない
- 逆に新着や無名論文だけでノイズ過多にしない
- 「安心して見られる論文」と「面白い掘り出し物」の両方を並べる
- 新規訪問者と継続訪問者の両方に刺さる棚を作る

### 16-2. イメージ
- **50%**: 被引用数が相対的に高いもの、注目度の高いもの、定番寄りのもの
- **50%**: 被引用数はまだ高くないが、新着・発想・タグ適合性の面で面白そうなもの

### 16-3. 運用上の扱い
- 候補取得段階では広めに拾う
- その後、公開候補を選ぶ段階で **高被引用枠** と **その他枠** に分ける
- 最終公開時に両者がだいたい半々になるよう調整する
- 厳密に 50:50 固定ではなく、日ごとの候補数や質に応じて多少のブレは許容する

### 16-4. 注意点
- 新着論文は構造上、被引用数が低くなりやすいので、単純な被引用数だけではなく公開日も併せて見る
- 「高被引用」だけでなく「よく参照されそう」「タグの代表例になりやすい」といった観点も使う
- 将来的には `高被引用 / 新着 / ランダムおすすめ` の3枠に分けてもよい

## 17. インフラ方針
### 17-1. 基本思想
- 管理の手軽さ重視
- 完全クラウド完結も可能
- 公開は静的配信中心
- 常時重いサーバーを持たない

### 17-2. 想定
- Cloudflare Pages / Workers 系
- APIで論文取得
- クラウドLLM APIで要約
- DBやJSON保存
- 静的生成で配信

### 17-3. コスト感
完全クラウド完結でも比較的安い想定。

目安:
- 節約寄り: 月300〜600円程度 + ドメイン相当
- 安定運用寄り: 月1,200〜1,500円程度 + ドメイン相当
- 少し余裕込み: 月1,500〜2,250円程度 + ドメイン相当

月2,000円前後でも十分現実的。

---

## 18. LLM方針
### 18-1. 役割
LLMは全文要約ではなく、**入口文の生成**に使う。

生成候補:
- 一言まとめ
- 3行まとめ
- 何が新しいか
- どんな人向けか
- 難易度
- 補助タグ

### 18-2. 方針
- 初期はクラウドLLMで十分
- 日次数十本規模なら小型クラウドLLMで回せる想定
- 無理にローカルLLMへ寄せなくてもよい
- 件数が増えたら後から再検討

---

## 19. 更新量の想定
現時点では以下の温度感がちょうどよい。

- 1日10本 → 月300本
- 1日20本 → 月600本
- 1日30本 → 月900本

推奨:
**月300〜600本公開、処理能力は月900本程度まで耐えられる設計**

含むもの:
- 論文取得
- 要約生成
- 承認
- 個別ページ

- 興味を引く見せ方
