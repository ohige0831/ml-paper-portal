# Claude Code 向け実装依頼文

## 前提
このプロジェクトは、**機械学習論文を日本語で“読ませる”のではなく、“開きたくさせる”入口サイト**を作るものです。

既存の仕様書・計画メモは同梱しているので、必ず読んだうえで、設計思想を崩さずに実装を進めてください。

同梱資料:
- `ml_paper_portal_implementation_spec_v1.md`
- `ml_paper_portal_spec_updated.md`
- `ml_paper_portal_plan.md`
- `cloudflare_openai_setup_checklist.md`

## ゴール
MVPとして、以下を一気通貫で動かしたいです。

1. OpenAlex から対象論文を取得する
2. 保存対象の論文を選定する
3. OpenAI API で短い日本語導入文を生成する
4. 下書き状態で保存する
5. 管理画面で承認できる
6. 承認済み論文から静的ページを生成する
7. Cloudflare Pages / Workers 前提でデプロイ可能な構成にする

## 絶対に守ってほしい方針

### 1. サイトの役割
- 全文翻訳サイトではない
- 長文解説サイトでもない
- **読む価値があるか・興味があるかを判断するための入口サイト**である

### 2. 生成文の役割
生成するのは以下のような短い入口文に限定する。
- 一言まとめ
- 3行まとめ
- キーワード
- 少し長めの要約
- こんな人向け
- 難易度

### 3. セキュリティ
- APIキーやトークンは**絶対にコードへ直書きしないこと**
- `OPENAI_API_KEY` と `CLOUDFLARE_API_TOKEN` は環境変数から読むこと
- `.env` は作らない、触らない、生成しない
- `.env.example` は必要なら作成してよい
- `.env` を `.gitignore` に含めること

### 4. 外部サービス前提
- Cloudflare を公開基盤として想定
- OpenAI は nano クラスの軽量利用を前提
- OpenAlex を主データソースとする
- PDF本体は保存しない
- 論文本文を転載しない

### 5. 設計思想
- **後からUI大改修しなくてよい骨格**を最初から作る
- 最初から全部盛りではなく、**骨格は完成形寄り・機能はMVP寄り**にする
- 普段は自走、変化点だけ人が介入する運用に寄せる

## 技術方針
具体的な言語やフレームワークは、以下を満たすなら提案してよいです。

### 希望
- TypeScript 優先でも Python 優先でも可
- ただし**Cloudflare Pages / Workers と相性がよく、保守しやすい構成**を優先
- 静的ページ生成しやすいこと
- 管理画面を軽く作りやすいこと

### 構成の方向性
以下に近い構成を想定しています。
- フロント: 静的サイト生成
- バックエンド: 取得・要約・承認フロー
- ストレージ: Cloudflare D1 / R2 / KV のどれをどう使うかは提案してほしい

## MVPに必須の機能

### A. 論文取得
- OpenAlex から機械学習系論文を取得
- Open Access 優先
- 取得対象は新着 + 被引用数多めを混ぜる
- **拾ってくる論文は、被引用数が多いものとその他を半々程度で混ぜる方針**

### B. 保存
最低限、以下の情報を保持できること。
- OpenAlex ID
- DOI
- 原題
- 公開日
- 被引用数
- OAリンク
- PDFリンク
- primary topic / topics
- 著者情報
- 日本語生成文
- タグ
- 状態フラグ

### C. 要約生成
OpenAI API で以下を生成すること。
- 日本語タイトル（必要なら）
- 一言まとめ
- 3行まとめ
- キーワード
- 少し長めの要約
- こんな人向け
- 難易度

### D. 承認フロー
状態管理の例:
- fetched
- summarized
- review_pending
- approved
- published
- error

管理画面では最低限以下ができること。
- 下書き一覧
- 論文情報確認
- 要約確認
- タグ確認
- 承認
- 差し戻し

### E. ページ生成
最低限以下のページが必要。
- トップページ
- 最新記事一覧
- 論文個別ページ
- タグ一覧ページ
- タグ個別ページ

### F. タグページ
タグページは以下を持つこと。
- タグタイトル
- 件数などの補助情報
- 「このタグとは？」の短い解説
- 解説は**トグルで開閉**できること
- **デフォルトは閉じる**
- その下に論文一覧を出す

## 内部リンク方針
- 本文中のモデル名や手法名は、**元祖論文へ自動で飛ばさない**
- 代わりに **タグページへ飛ばす**
- 例: `ViT` → `ViT` タグページ
- 「元祖論文」と断定する機能は入れない
- 必要ならタグページ内で「代表的な論文」「まず見るなら」を出す形にする

## トップページ方針
- 高度なレコメンドは不要
- 最低限ほしいもの:
  - 今日のおすすめ
  - 最新記事
  - 人気タグ
- 「今日のおすすめ」は**ランダムピック寄り**でよい
- ただし極端な偏りを避ける軽い制御があってよい

## URL設計
できれば以下に近い素直な構造にしたいです。
- `/papers/<slug>`
- `/tags/<slug>`
- `/latest`
- `/recommended`

必要なら、最適な案を提案してください。

## タグ設計ルール
タグは階層っぽく見せたいです。
ただし内部的には柔軟な多対多でよいです。

ルール:
- 大タグ = 分野 / タスク
- 中タグ = モデル / 手法群
- 小タグ = 技術要素 / 特徴

例:
- 大タグ: 画像認識, LLM, 超解像
- 中タグ: ViT, Diffusion, Transformer
- 小タグ: 軽量化, 蒸留, 量子化

## やらないこと
初期段階では以下を入れないでください。
- 全文翻訳
- 元祖論文自動推定
- 高度なおすすめアルゴリズム
- コメント機能
- ユーザー投稿機能
- PDF本体保存
- 過度に複雑な検索UI
