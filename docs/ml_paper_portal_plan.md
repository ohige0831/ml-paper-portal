# 機械学習論文 日本語導入サイト 企画メモ

## 1. 企画の核

### コンセプト
英語中心で流量の多い機械学習論文を、**日本語の短い入口情報**でパラパラ眺められるようにするサイトを作る。

このサイトの目的は、論文の全文を代替することではない。
あくまで、

- 面白そうか
- 自分の興味に触れるか
- 読みに行く価値がありそうか

を日本語で素早く判断できるようにすることを目指す。

### 目指す体験
- 辞書や雑誌をぱらぱら眺める感覚で論文を見られる
- 有用性を先に断定するのではなく、興味が湧くかどうかで次の行動を決められる
- 気になった論文は、そのまま元論文PDFや論文ページへ飛べる
- 読み込みはユーザー自身がLLMや翻訳ツールに任せればよい

### この企画が強い理由
- 機械学習系は論文の更新頻度が高く、定期的に見に来る理由がある
- 英語論文が多く、日本語の入口整備に価値がある
- 全文要約ではなく短文導入に絞ることで、運用コストとトークンコストを抑えやすい
- サイトを見るだけなので、配布ソフトのようなユーザー依存バグ対応がほぼない
- サポートよりも、運用保守と見せ方改善に集中できる

---

## 2. サイトの立ち位置

### これは何ではないか
- 論文の全文和訳サイトではない
- 論文レビューサイトではない
- 厳密な有用論文選別サイトではない
- 論文の代替物ではない

### これは何か
- **論文ブラウジングの入口**
- **興味のフックを見つける場所**
- **読む前の判断レイヤー**
- **日本語の帯・あらすじを付ける場所**

### 一言でいうと
**論文を評価する場所ではなく、論文に惚れるきっかけを作る場所**

---

## 3. 対象分野

### 最初の対象
機械学習系に絞る。

### 理由
- 進歩が速く、新着を見に来る理由がある
- 「何か面白いものはないか」とパラパラ見たい需要が強い
- 英語圏では論文閲覧文化が強いが、日本語の軽い入口はまだ余地がある
- タグ設計と相性が良い

### 想定タグ例
- LLM
- Vision
- Multimodal
- Agent
- RL
- GNN
- Time Series
- Medical AI
- Optimization
- Interpretability

最初から広げすぎず、機械学習系の中でも見やすい分類を優先する。

---

## 4. データ源

### 第一候補
**OpenAlex**

### 理由
- works / authors / institutions / topics を扱える
- 被引用数、トピック、Open Access 情報などが揃っている
- 新着取得、タグ化、最新順、被引用数順と相性がよい
- APIが整備されている

### 補助候補
**arXiv**

### 役割
- プレプリント導線の補強
- 機械学習系カテゴリの補助

### 基本方針
最初は **OpenAlex主軸、必要に応じてarXiv補助** で始める。
最初から依存先を増やしすぎない。

## 5. 1論文あたりの表示情報

### 最小構成
- 日本語タイトル
- 一言要約
- 3行要約
- 何が新しい？
- こんな人向け
- 難易度
- タグ
- 元論文リンク

### 方針
長文を出しすぎない。
あくまで「読むかどうかの判断材料」を短く整える。

### 理由
- 一覧性が高い
- パラパラ見しやすい
- トークンコストを抑えられる
- 要約品質のぶれを抑えやすい
- 日次運用しやすい

## 7. システム構成方針

### 最重要方針
**管理の手軽さと自動走行のしやすさを優先する。**

### 構成の方向性
- 公開サイトは極力静的にする
- 定期バッチで新着取得・要約生成・ページ生成を行う
- 生成後は静的配信する
- 普段は自動、例外時だけ人が見る

### 想定構成
- OpenAlex API で論文取得
- クラウドLLM API で短文要約生成
- DB または JSON 保存
- 静的ページ生成
- CDN/静的ホスティングで公開

### 推奨イメージ
- Cloudflare Pages / Workers 系で公開と定期実行
- OpenAlex で取得
- クラウドLLM API で要約
- 承認後に静的生成して公開

---

## 8. 運用フロー

### 初期フェーズ
1. OpenAlex から候補論文を取得
2. 要約とタグを生成
3. 管理画面で内容を確認
4. 承認したものだけ公開

### 安定化フェーズ
- 基本は自動取得
- 基本は自動要約
- 怪しいものだけ確認
- 問題ないものは自動公開寄りへ

### 最終的な理想
- 収集: 自動
- 要約生成: 自動
- ページ生成: 自動
- デプロイ: 自動
- 失敗時のみ通知
- 人間は変化点だけ見る

### 日常運用のイメージ
**普段は寝ているが、朝には新しい論文カードが棚に並んでいて、変化が起きた時だけ管理者が出てくるサイト**

---

## 9. 月間本数の想定

### 現実的な想定
最初は以下の規模感がちょうどよい。

- 候補取得: 月3,000〜9,000件程度
- 要約生成: 月300〜900件程度
- 公開本数: 月300〜600件程度を目安

### 目安
- 1日10本 = 月300本
- 1日20本 = 月600本
- 1日30本 = 月900本

### 推奨ライン
**月300〜600本公開想定、処理能力としては月900本まで耐えられる設計**

### 理由
- 量を増やしすぎると品質管理が難しくなる
- サイトの一覧性が落ちる
- 面白くない論文の混入率が上がる
- 最初は量より見せ方の精度が大事

---

## 10. コスト感

### 方針
- 公開サイトのコストはなるべく軽くする
- LLMコストは短文生成に限定する
- 完全クラウド完結でも小さく始める

### 完全クラウド完結の想定
- 静的ホスティング + 定期実行
- OpenAlex取得
- クラウドLLM API要約
- 静的ページ生成

### ざっくり費用感
1ドル150円換算時の目安:

- $2〜$4/月 = 約300〜600円/月
- $5/月 = 約750円/月
- $8〜$10/月 = 約1,200〜1,500円/月
- $10〜$15/月 = 約1,500〜2,250円/月

### 現実的な見立て
- 節約寄り: 月300〜600円 + ドメイン相当
- 安定運用寄り: 月1,200〜1,500円 + ドメイン相当
- 少し余裕を見ても: 月1,500〜2,250円 + ドメイン相当

### 結論
**月2,000円前後でもかなり現実的**

---

## 11. LLM運用方針

### 方針
最初はクラウドLLMを使う前提でもよい。

### 理由
- 日次数十本規模ならコストが重くなりにくい
- ローカルLLM運用の保守を後回しにできる
- 品質が比較的安定する
- 最初の構築を素早く進められる

### 将来の拡張余地
必要なら後から
- ローカルLLMで下書き
- クラウドLLMで再整形

のようなハイブリッド運用にも移行できる。

### 現時点でのおすすめ
**最初は小規模クラウドLLMで十分**

---

## 12. 収支イメージ

### 強み
- 固定費がかなり軽い
- ソフト配布型よりサポートコストが低い
- ページが資産として積み上がる
- 広告収益モデルと相性がよい

### 収支の考え方
最初から大きく稼ぐというより、

- ページを積み上げる
- 検索流入を増やす
- タグ回遊を作る
- 面白い入口として定着させる

ことで、低コストのまま黒字化を狙う。

### 結論
**流入さえ作れれば、完全クラウド完結でも収支プラスは十分狙える。**

---

## 14. この案の本質

### 核心
これは「論文の代替サイト」ではない。

これは、
**日本語の入口を作ることで、興味を起点に論文へ手を伸ばしやすくするサイト** である。

### 価値の出し方
- 正しさで選ばせるのではなく、好奇心でめくらせる
- 有用性を断定するのではなく、興味のフックを渡す
- すべてを説明するのではなく、読む前の判断材料を整える

### 一言まとめ
**論文を評価する場ではなく、論文に出会う場を日本語で作る。**
