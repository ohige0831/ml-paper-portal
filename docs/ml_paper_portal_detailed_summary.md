# ML Paper Portal ここまでの詳細まとめ

## 1. プロジェクトの目的

ML Paper Portal は、**機械学習論文を日本語で“読ませる”のではなく、“開きたくさせる”ための入口サイト**として設計・実装を進めているプロジェクトです。

目的は以下です。

- 英語論文を読む前に、日本語でざっくり全体像を掴めるようにする
- 「読む価値があるか」「興味があるか」を短時間で判断できるようにする
- 気になった論文は、そのまま元論文へ飛べるようにする
- タグや一覧を通じて、論文を“パラパラ眺める”ような体験を作る
- 完全な百科事典や全文翻訳サイトではなく、**理解の入口**として機能させる

この思想により、サイト全体は以下の方向で固めている。

- 文章は長すぎず短すぎない
- 背景・手法・結果がつかめる粒度にする
- 詳しい説明は必要以上に抱え込まない
- 深掘りは元論文や外部情報へ逃がす
- タグや一覧を含めて、回遊しやすい構造にする

---

## 2. 企画段階で固めた思想

当初の検討では、以下のような方針を固めた。

### 2.1 なぜこのテーマか
- 機械学習は今後も伸び続ける可能性が高い分野
- 論文・モデル・手法・タグなど、検索入口が非常に多い
- 英語論文は読む前の心理的コストが高い
- 日本語で概要だけでもわかると「読むかどうか」の判断がしやすい
- 機械学習が盛り上がるほど、入口サイトの価値も上がりやすい

### 2.2 サイトの立ち位置
- 全文翻訳サイトではない
- 用語辞典サイトでもない
- 元祖論文の自動断定はしない
- 代わりに「代表的」「まず見るなら」程度の導線に留める
- 興味のフックを作り、次の検索や元論文閲覧に自然に繋げる

### 2.3 コンテンツの基本構成
論文ページは以下のような情報粒度を持たせる方針にした。

- 日本語タイトル
- 英語原題
- 一言説明
- 3行まとめ
- キーワード
- もう少しだけ中身を見る
- こんな人向け
- 難易度
- 元論文情報
- タグ

この構成により、読む側は
1. 何の論文か
2. なぜ重要か
3. 何をしたのか
4. どういう人向けか
を短時間で把握できる。

## 3. 技術構成の決定

議論の結果、以下の構成を採用した。

### 3.1 基本構成
- **Cloudflare Pages**: 公開サイト
- **Cloudflare Workers**: 管理画面、管理API、cron、補助処理
- **Cloudflare D1**: 論文メタデータ、要約、タグ、状態管理
- **OpenAI API (GPT-5.4 nano)**: 日本語要約生成
- **OpenAlex**: 論文取得元
- **GitHub Actions**: OpenAlex の定時取得（429回避策として導入）
- **GitHub**: コード管理と Actions 実行基盤

### 3.2 Secrets 管理方針
API キーやトークンは Claude Code に直接触らせず、ユーザー側で `.env` と Cloudflare / GitHub Secrets により管理する方針とした。

利用する主要な環境変数:
- `OPENAI_API_KEY`
- `CLOUDFLARE_API_TOKEN`
- `ADMIN_PASSWORD`
- `INGEST_TOKEN`
- `WORKER_URL`

`.env` は Git に載せず、`.env.example` のみコミットする方針を採用した。

### 3.3 コスト感
- 初期は Cloudflare 側はほぼ無料で運用できる可能性が高い
- 主な変動費は OpenAI API
- GPT-5.4 nano は安価で、短文要約用途と相性が良い
- 最初は月 $5〜$10 程度で十分試せる見込み
- 初月だけ少し厚めに回して記事棚を増やす案も有効

- orphan branch を使って履歴をきれいに作り直し、push 成功

## 7. UI と UX の改善

実装後、いくつかの改善を追加で行った。

### 7.1 英語原題の表示
当初は日本語タイトルだけで、正式な論文名が見えにくかったため、以下を追加した。

- **論文カード**  
  日本語タイトルの下に、英語原題を小さくグレーで表示
- **個別ページ**  
  `<h1>` 直下に英語原題を表示

これにより、「何の論文か」が一瞬で把握しやすくなった。

### 7.2 スマホ版余白の改善
論文個別ページだけスマホ表示時の左右余白が少なく、本文が画面端に近く見えたため修正した。

- モバイル時の `.paper-detail` の左右 padding を拡大
- グローバルの `.container` は変えず、論文ページだけ調整
- 結果として、スマホ版もかなり自然な見え方になった

### 7.3 タグページの評価
タグページは、
- 一言説明
- 論文件数
- すぐ下に論文カード
という構成で、非常に使いやすいと感じられた。

この粒度がちょうどよく、
- 説明しすぎない
- 齟齬が少ない
- すぐ論文を見る導線がある
という点で、今後もこの方向性を維持する価値が高い。

## 8. 管理画面の実装と改善

管理画面は Worker 側に実装された。

### 8.1 管理画面の機能
- ステータス別一覧
- レビュー待ち論文の確認
- 右側で詳細を閲覧
- 3行まとめ

- 要約
- こんな人向け
- 難易度
- 元論文情報
- **承認して公開**
- **差し戻し**

この画面は想像以上に出来が良く、運用に必要な情報が揃っていた。

### 8.2 認証問題
一時 `/admin` にアクセスすると `{"error":"Unauthorized"}` が返る問題があった。

原因は2つだった。

1. ingest 用 Bearer 認証ミドルウェアのスコープが広すぎて `/admin` を先に食っていた
2. Basic 認証で、パスワードに `:` が含まれる場合の分割バグがあった

修正後、
- `/admin` でブラウザの Basic 認証ダイアログが出るようになり
- `admin` + 新しい `ADMIN_PASSWORD` でログイン可能になった

## 9. OpenAlex 429 問題と恒久対策

Cloudflare KIX（大阪）からの OpenAlex アクセスが継続的に 429 を返す問題が発生した。

### 9.1 問題の内容
- Worker 側からの fetch がブロックされる
- `mailto=` パラメータを付けても改善しない
- IP レベルで弾かれている可能性が高い

### 9.2 検討した案
- Cloudflare Smart Placement
- Semantic Scholar への乗り換え
- ローカル取得
- **GitHub Actions + Worker ingest**

### 9.3 採用した案
**GitHub Actions cron + Worker ingest endpoint** を採用した。

構成:
1. GitHub Actions が OpenAlex にアクセス
2. 取得した論文情報を Worker の `/api/ingest` へ POST
3. Worker が D1 に登録
4. summarize cron や管理画面から要約へ進む

この構成により、Cloudflare の発信元 IP 問題を回避した。

### 9.4 追加実装
- `worker/src/routes/ingest.ts`
- `worker/src/lib/openalex.ts`
- `scripts/fetch_openalex.js`
- `.github/workflows/daily_fetch.yml`

さらにフィルターも改善した。

- `concepts.id:C41008148` から
- `primary_topic.field.id:17`（Computer Science）へ変更

## 10. GitHub Actions の導入と成功

### 10.1 初回失敗
最初の GitHub Actions 実行は失敗した。
原因は、Cloudflare Worker 側が古いデプロイのままで、新しく追加した ingest endpoint が反映されていなかったこと。

### 10.2 再デプロイ後
worker / web を再デプロイしたところ、GitHub Actions は成功した。

これにより、
- OpenAlex 取得
- GitHub Actions 実行
- Worker ingest
- D1 反映

が通った。

## 16. 今後の推奨方針

### 最優先
- 半自動運用をしばらく継続し、流れてくる論文の質を見る
- review_pending を数本単位で捌き、公開基準を掴む
- GitHub Actions の安定確認を続ける

### その次
- Search Console / sitemap 導入
- タグや一覧の品質向上
- 初月の予算と本数の最適化
- 手動論文投入機能の検討

### 基本方針
- 盛りすぎない
- 説明しすぎない
- 入口としての価値を壊さない
- 軽く、わかりやすく、次へ進みやすく保つ

**自動取得・自動要約・人間による承認・公開まで通る、実運用可能な半自動メディアの骨格**

**どう育てるか、どう磨くか**
